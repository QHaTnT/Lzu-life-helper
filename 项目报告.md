# 兰州大学生活助手 - 项目报告

**项目名称**：兰州大学生活助手（Lzu-life-helper）

**开发周期**：2025年10月 ～ 2025年12月

**项目类型**：前后端分离 Web 应用

**GitHub**：https://github.com/QHaTnT/Lzu-life-helper

## 一、项目背景

兰州大学师生在日常校园生活中面临信息分散的问题：二手物品交易依赖微信群、场馆预约需要线下排队、校车班次靠口耳相传、校园活动信息零散。缺乏统一的数字化服务入口，导致信息获取效率低、服务体验差。

本项目目标是构建一个面向兰州大学师生的校园综合服务平台，将二手交易、场馆预约、出行查询、社区互动、活动报名等高频需求整合到一个 Web 应用中。

**核心价值：**

- 信息聚合：五大模块统一入口，告别信息碎片化
- 零门槛部署：SQLite 零配置数据库，一键脚本启动，无需服务器
- AI 辅助开发：全程借助 AI 完成架构设计、代码实现、Bug 修复与文档撰写

## 二、功能模块

共 6 大功能模块，13 张数据表，13 个前端路由页面。

| 模块 | 核心功能 | 数据表 |
|------|---------|--------|
| 用户系统 | 注册、登录、JWT 鉴权、个人资料管理 | users |
| 二手市场 | 发布/搜索/筛选商品，留言议价，联系卖家 | products, product_comments |
| 场馆预约 | 查看时段，在线预约，防重复预约 | venues, venue_time_slots, bookings |
| 校车服务 | 查看城关↔榆中班次时刻表 | bus_routes, bus_schedules |
| 生活圈 | 发布动态/活动，点赞，评论，图片上传 | posts, post_comments, likes |
| 校园活动 | 发布活动，在线报名，查看报名名单 | activities, activity_registrations |

### 2.1 用户系统

- 学号/用户名注册，bcrypt 密码哈希存储
- JWT 双 Token 机制：Access Token（30分钟）+ Refresh Token（7天）
- 路由守卫：未登录自动跳转登录页，已登录跳过登录页
- 个人中心：修改资料、查看我的商品/动态/活动

### 2.2 二手市场

- 商品列表支持分类筛选（6类）、关键词搜索、价格区间过滤
- 发布商品：标题/描述/价格/分类/图片（最多9张）
- 商品详情：图片轮播、卖家信息（含手机号）、留言板
- "联系卖家"显示手机号，"我想要"自动滚动到留言区并聚焦输入框

### 2.3 场馆预约

- 展示未来3天可预约时段，实时显示剩余名额
- 预约时检查容量，防止超额预约
- 支持取消预约，已预约数量同步回滚

### 2.4 校车服务

- 城关→榆中、榆中→城关双向路线
- 展示发车时间、座位数、是否仅工作日运营

### 2.5 生活圈

- 动态分类：失物招领/吐槽/问答互助/分享/活动/其他
- 朋友圈式交互：点赞（防重复）、评论、浏览数统计
- 图片上传：最多9张，支持预览和删除
- 发布活动：独立表单，含主办方/地点/时间/人数限制

### 2.6 校园活动

- 活动列表展示，支持报名/取消报名
- 防重复报名：数据库唯一约束 (activity_id, user_id)
- 发布者可查看报名名单，参与人数实时更新

## 四、核心技术实现

### 4.1 统一响应封装

后端所有接口统一返回 `{code, msg, data}` 格式，前端 Axios 拦截器自动解包，调用方直接拿到 data，无需每次判断外层结构。

后端返回格式：

```json
{
  "code": 200,
  "msg": "success",
  "data": { }
}
```

前端拦截器自动解包，调用方直接使用：

```javascript
// 拦截器
response.interceptors.response.use(res => res.data.data)

// 调用方无需处理外层
const products = await productAPI.getProducts()
```

### 4.2 JWT 鉴权流程

```
登录
  → 后端签发 Access Token（30分钟）
  → 前端存入 localStorage
  → Axios 拦截器每次请求自动附加 Authorization: Bearer <token>
  → 后端 deps.py 解析 Token → 查询用户 → 注入到路由函数
  → Token 过期 → 前端拦截 401 → 跳转登录页
```

### 4.3 SQLite 兼容适配

项目初期设计为 MySQL，本地开发切换为 SQLite 时处理了两个兼容性问题：

**问题1**：SQLite 不支持连接池的 pool_recycle 参数

```python
_is_sqlite = settings.DATABASE_URL.startswith("sqlite")
if _is_sqlite:
    engine = create_engine(url, connect_args={"check_same_thread": False})
else:
    engine = create_engine(url, pool_pre_ping=True, pool_recycle=3600)
```

**问题2**：SQLite 不支持行级锁 with_for_update()

```python
if not _is_sqlite:
    slot = db.query(VenueTimeSlot).filter(...).with_for_update().first()
else:
    slot = db.query(VenueTimeSlot).filter(...).first()
```

### 4.4 防重复操作设计

点赞、报名等操作通过数据库唯一约束从根本上防止重复，比应用层判断更可靠：

```python
# 防重复点赞
class Like(Base):
    __table_args__ = (
        UniqueConstraint("user_id", "post_id", name="uq_like_user_post"),
    )

# 防重复报名
class ActivityRegistration(Base):
    __table_args__ = (
        UniqueConstraint("activity_id", "user_id", name="uq_activity_user_registration"),
    )
```

### 4.5 文件上传流程

```
前端选择图片
  → FormData 封装
  → POST /api/v1/upload
  → 后端 Pillow 验证图片格式
  → 保存到 uploads/ 目录
  → 返回可访问的静态 URL
  → 前端存储 URL，发布时随表单一起提交
```

### 4.6 共享底部导航组件

6个主页面共用 BottomNav.vue 组件，通过 useRoute() 自动高亮当前页签：

```javascript
const navItems = [
```

## 五、数据库设计

### 5.1 表关系概览

```
users
  ├── products ── product_comments
  ├── bookings ── venue_time_slots ── venues
  ├── posts ── post_comments
  ├── posts ── likes
  └── activities ── activity_registrations

bus_routes ── bus_schedules（独立，无用户关联）
```

### 5.2 设计原则

- **全关联建模**：所有业务表均持有 user_id 外键，支持"我的xxx"查询
- **级联删除**：cascade="all,delete-orphan"，删除用户自动清理关联数据
- **唯一约束防重**：点赞、报名在数据库层面保证幂等性
- **状态字段**：商品有 status（available/sold/removed），预约有 status（pending/confirmed/cancelled/completed）

### 6.2 典型 Bug 修复案例

**案例一：JWT 鉴权失效（所有需要登录的操作返回 401）**

现象：登录成功后，留言、预约等操作全部返回 401 未授权。

根因 A：Axios 拦截器中调用 useAuthStore() 时 Pinia 尚未初始化，读不到 Token。

修复：改为直接读取 localStorage.getItem('token')，绕过 Pinia 初始化时序问题。

根因 B：python-jose 解码 JWT 后 sub 字段为字符串 "1"，数据库查询 User.id == "1" 类型不匹配，查不到用户。

修复：后端 deps.py 加 int(user_id) 强制类型转换。

---

**案例二：图片上传路由 404**

现象：上传图片返回 404，实际请求路径变成 /upload/upload。

根因：路由注册时 prefix 设为 /upload，handler 装饰器又写了 /upload，导致路径双重叠加。

修复：__init__.py 中 prefix 改为空字符串，handler 保留 /upload 路径。

---

**案例三：本地无 MySQL 环境无法启动**

现象：团队成员本地未安装 MySQL，项目完全无法运行。

解决：将数据库切换为 SQLite，同时处理 SQLite 不支持 pool_recycle 和 with_for_update() 的兼容性问题，实现零配置本地开发。
