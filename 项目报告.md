# 兰州大学生活助手 - 项目报告

**项目名称**：兰州大学生活助手（Lzu-life-helper）

**开发周期**：2025年10月 ～ 2025年12月

**项目类型**：前后端分离 Web 应用

**GitHub**：https://github.com/QHaTnT/Lzu-life-helper

## 一、项目背景与目标

### 1.1 背景

兰州大学师生在日常校园生活中面临信息分散的问题：二手物品交易依赖微信群、场馆预约需要线下排队、校车班次靠口耳相传、校园活动信息零散。缺乏一个统一的数字化服务入口，导致信息获取效率低、服务体验差。

### 1.2 目标

构建一个面向兰州大学师生的校园综合服务平台，将二手交易、场馆预约、出行查询、社区互动、活动报名等高频需求整合到一个 Web 应用中，提供统一的用户体验。

### 1.3 核心价值

- **信息聚合**：五大模块统一入口，告别信息碎片化
- **零门槛部署**：Docker Compose 一条命令启动，无需手动配置任何环境
- **AI 辅助开发**：全程借助 AI 完成架构设计、代码实现、Bug 修复与文档撰写，显著压缩开发周期

### 2.2 功能详述

#### 用户系统
- 学号/用户名注册，bcrypt 密码哈希存储
- JWT 双 Token 机制：Access Token（30分钟）+ Refresh Token（7天）
- 路由守卫：未登录自动跳转登录页，已登录跳过登录页
- 个人中心：修改资料、查看我的商品/动态/活动

#### 二手市场
- 商品列表支持分类筛选（6类）、关键词搜索、价格区间过滤
- 发布商品：标题/描述/价格/分类/图片（最多9张）
- 商品详情：图片轮播、卖家信息（含手机号）、留言板
- "联系卖家"显示手机号，"我想要"自动滚动到留言区并聚焦输入框

#### 场馆预约
- 展示未来3天可预约时段，实时显示剩余名额
- 预约时检查容量，防止超额预约
- 支持取消预约，booked_count 同步回滚

#### 校车服务
- 城关→榆中、榆中→城关双向路线
- 展示发车时间、座位数、是否仅工作日运营

#### 生活圈
- 动态分类：失物招领/吐槽/问答互助/分享/活动/其他
- 朋友圈式交互：点赞（防重复）、评论、浏览数统计
- 图片上传：最多9张，支持预览和删除
- 发布活动：独立表单，含主办方/地点/时间/人数限制

#### 校园活动
- 活动列表展示，支持报名/取消报名
- 防重复报名：数据库唯一约束 `(activity_id, user_id)`
- 发布者可查看报名名单，参与人数实时更新

## 四、核心技术实现

### 4.1 统一响应封装

后端所有接口统一返回 `{code, msg, data}` 格式，前端 Axios 拦截器自动解包，调用方直接拿到 `data`，无需每次判断外层结构。

```python
# 后端统一响应
def success(data=None, msg="success"):
    return {"code": 200, "msg": msg, "data": data}
```

```javascript
// 前端拦截器自动解包
instance.interceptors.response.use(response => {
    const { code, msg, data } = response.data
    if (code === 200) return data
    if (code === 401) { router.push('/login'); return }
    return Promise.reject(new Error(msg))
})

// 调用方直接拿数据
const products = await productAPI.getProducts()  // 直接是数组
```

### 4.2 JWT 鉴权流程

```
登录 → 后端签发 Access Token（30min）
     → 前端存入 localStorage
     → Axios 拦截器每次请求自动附加 Authorization: Bearer <token>
     → 后端 deps.py 解析 Token → 查询用户 → 注入到路由函数
     → Token 过期 → 前端拦截 401 → 跳转登录页
```

**后端 Token 解析关键代码：**

```python
def get_current_user(token: str = Depends(oauth2_scheme), db: Session = Depends(get_db)):
    payload = jwt.decode(token, settings.JWT_SECRET_KEY, algorithms=["HS256"])
    user_id = int(payload.get("sub"))  # 强制转 int，防止类型不匹配
    user = db.query(User).filter(User.id == user_id).first()
    return user
```

### 4.3 场馆预约并发安全（双重锁机制）

这是项目中技术难度最高的模块，采用 Redis 分布式锁 + MySQL 行锁双重保障：

```
用户请求预约
  → Redis 分布式锁（SET NX EX，防止并发请求同时进入）
  → MySQL SELECT ... FOR UPDATE 行锁（防止并发写）
  → 检查剩余容量
  → 写入预约记录，更新计数
  → 释放 Redis 锁
```

**Redis 分布式锁：**

```python
lock_key = f"booking_lock:{time_slot_id}"
lock_acquired = redis_client.set(lock_key, lock_value, nx=True, ex=lock_timeout)
if not lock_acquired:
    return None  # 有人正在预约，拒绝
```

**MySQL 行锁：**

```python
time_slot = db.query(VenueTimeSlot).filter(
    VenueTimeSlot.id == time_slot_id
).with_for_update().first()  # 锁住这一行，其他事务等待
```

**数据库唯一约束兜底：**

```python
class Booking(Base):
    __table_args__ = (
        UniqueConstraint("user_id", "time_slot_id", name="uq_booking_user_slot"),
    )
```

### 4.4 防重复操作设计

点赞、报名通过数据库唯一约束从根本上防止重复：

```python
# 防重复点赞
class Like(Base):

# 防重复报名
class ActivityRegistration(Base):
```

## 五、数据库设计

### 5.1 ER 关系概览

```
users ──┬── products ── product_comments
        ├── bookings ── venue_time_slots ── venues
        ├── posts ──┬── post_comments
        │           └── likes
        ├── activities ── activity_registrations
        └── (bus_routes ── bus_schedules，无用户关联)
```

### 5.2 设计原则

- **全关联建模**：所有业务表均持有 `user_id` 外键，支持"我的xxx"查询
- **级联删除**：`cascade="all,delete-orphan"`，删除用户自动清理关联数据
- **唯一约束防重**：点赞、报名在数据库层面保证幂等性
- **状态字段**：商品有 `status`（available/sold/removed），预约有 `status`（pending/confirmed/cancelled/completed）

### 5.3 种子数据

|---------|------|

| 场馆 | 4个，未来3天×4时段 = 48个可预约时段 |
| 校车班次 | 12条（城关↔榆中双向各6班） |

### 7.2 典型 Bug 修复案例

**案例一：JWT 鉴权失效（所有操作返回 401）**

- **现象**：登录成功后，留言、预约等操作全部返回 401
- **根因 A**：前端 Axios 拦截器中 `useAuthStore()` 在 Pinia 未初始化时调用，读不到 Token
- **修复**：改为直接读 `localStorage.getItem('token')`
- **根因 B**：JWT `sub` 字段为字符串 `"1"`，数据库查询 `User.id == "1"` 类型不匹配
- **修复**：后端加 `int(user_id)` 强制类型转换

**案例二：图片上传路由 404**

- **现象**：上传图片返回 404，实际路径变成 `/upload/upload`
- **根因**：路由注册时 prefix 和 handler 路径双重叠加
- **修复**：prefix 置空，handler 保留 `/upload`

**案例三：Docker 国内网络问题**

- **现象**：`apt-get` 报 502 Bad Gateway，无法构建后端镜像
- **根因**：Debian 官方源在国内不通
- **修复**：Dockerfile 中配置清华 Debian 镜像源
