## 一、项目概述

### 1.1 项目背景

兰州大学师生在日常校园生活中面临信息分散的问题：二手物品交易依赖微信群、场馆预约需要线下排队、校车班次靠口耳相传、校园活动信息零散。缺乏一个统一的数字化服务入口。

本项目构建了一个面向兰州大学师生的校园综合服务平台，将高频校园需求整合到一个 Web 应用中。

### 3.3 设计亮点

**全关联建模**：所有业务表均持有 `user_id` 外键，支持"我的商品""我的动态""我的活动"等个人中心查询。

**唯一约束防重**：

```python
# 点赞防重复
class Like(Base):
    __table_args__ = (
        UniqueConstraint("user_id", "post_id", name="uq_like_user_post"),
    )

# 报名防重复
class ActivityRegistration(Base):
    __table_args__ = (
        UniqueConstraint("activity_id", "user_id", name="uq_activity_user_registration"),
    )
```

**级联删除**：删除用户时自动清理关联数据。

```python
products = relationship("Product", back_populates="seller", cascade="all,delete-orphan")
```

## 四、核心功能详解

### 4.1 JWT 鉴权流程

**流程图：**

```
用户输入账号密码
    → 前端 POST /api/v1/auth/login
    → 后端验证密码（bcrypt）
    → 签发 JWT Token（sub=user_id, exp=30min）
    → 返回 {access_token, token_type}
    → 前端存入 localStorage
    → 后续请求自动附加 Authorization: Bearer <token>
    → 后端 deps.py 解析 Token → 查询用户 → 注入路由函数
    → Token 过期 → 前端拦截 401 → 跳转登录页
```

<!-- 插入图片：登录页面截图 -->

**关键代码（Axios 拦截器自动注入 Token）：**

```javascript
// frontend/src/api/request.js
const token = localStorage.getItem('token')
if (token) {
    config.headers.Authorization = `Bearer ${token}`
}
```

**关键代码（后端 Token 解析）：**

```python
# backend/app/api/deps.py
def get_current_user(token: str = Depends(oauth2_scheme), db: Session = Depends(get_db)):
    payload = jwt.decode(token, settings.JWT_SECRET_KEY, algorithms=["HS256"])
    user_id = int(payload.get("sub"))  # 强制转 int，防止类型不匹配
    user = db.query(User).filter(User.id == user_id).first()
    return user
```

### 4.3 场馆预约并发安全

这是项目中技术难度最高的模块，采用**双重锁机制**防止并发超额预约：

**架构：**

```
用户请求预约
    → Redis 分布式锁（SET NX EX，防止并发请求同时进入）
    → MySQL SELECT ... FOR UPDATE 行锁（防止并发写）
    → 检查剩余容量
    → 写入预约记录，更新计数
    → 释放 Redis 锁
```

**Redis 分布式锁：**

```python
lock_key = f"booking_lock:{time_slot_id}"
lock_acquired = redis_client.set(lock_key, lock_value, nx=True, ex=lock_timeout)
if not lock_acquired:
    return None  # 有人正在预约，拒绝
```

**MySQL 行锁：**

```python
time_slot = db.query(VenueTimeSlot).filter(
    VenueTimeSlot.id == time_slot_id
).with_for_update().first()  # 锁住这一行，其他事务等待
```

**数据库唯一约束兜底：**

```python
class Booking(Base):
    __table_args__ = (
        UniqueConstraint("user_id", "time_slot_id", name="uq_booking_user_slot"),
    )
```

### 6.2 典型 Bug 修复案例

**案例一：JWT 鉴权失效**

- **现象**：登录成功后所有操作返回 401
- **根因**：前端 Pinia 未初始化读不到 Token + 后端 JWT sub 字段类型不匹配
- **修复**：前端改读 localStorage + 后端加 `int()` 强制转换

**案例二：图片上传路由 404**

- **现象**：上传返回 404，路径变成 `/upload/upload`
- **根因**：路由注册时 prefix 和 handler 路径双重叠加
- **修复**：prefix 置空，handler 保留 `/upload`

## 十、常见面试问题

**Q: 为什么选择 FastAPI 而不是 Django/Flask？**
A: FastAPI 基于异步框架 Starlette，性能优于 Flask；自动生成 Swagger 文档，前后端协作更高效；原生支持 Pydantic 数据校验，代码更简洁。

**Q: 为什么用 MySQL 而不是 MongoDB？**
A: 项目涉及多表关联查询（用户-商品-评论、场馆-时段-预约），关系型数据库更适合。MongoDB 适合文档型数据，不适合这种强关联场景。

**Q: 场馆预约如何防止并发超额？**
A: 三层防护：Redis 分布式锁控制并发请求入口 → MySQL 行锁保证数据一致性 → 数据库唯一约束兜底防止重复预约。

**Q: 前后端如何联调？**
A: 前端 Axios 拦截器自动注入 JWT Token；后端统一响应封装 `{code, msg, data}`；Nginx 反向代理解决跨域；Swagger 文档自动生成，前后端可独立开发。

**Q: Docker 部署的优势？**
A: 四容器编排（MySQL + Redis + Backend + Frontend），一条命令启动整个系统，环境一致性有保障，团队成员无需手动配置任何依赖。
